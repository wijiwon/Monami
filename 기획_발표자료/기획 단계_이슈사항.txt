# rooms 테이블에 drawing_end가 존재해야 할 듯?
라운드가 시작되면 모든 유저가 그림을 다 그리거나 제한된 시간이 종료되면 다음 판으로 넘어가는데 이 그림을 다 그린 유저를 동시에 표현하기 위해서는 컬럼을 하나 만들어야 하지 않을까?
그림을 다 그리면 해당 컬럼에 중복방지를 거쳐 유저의 정보를 넣고, 이 컬럼의 유저 수와, 해당 게임방에 들어와있는 전체 유저 수를 비교하여 모든인원이 완료되면 다음 라운드로 진행하고 라운드로 넘어가면서 해당 컬럼의 값들을 비운다.
이러면 게임을 진행하면서 좌측 상단에 뜨는 완료한 유저수를 표현할 수 있을 것으로 생각된다. 


# post 테이블에 likes, views 컬럼이 빠져있다.
해당 게시글의 좋아요 수를 표현할 likes, views 컬럼이 빠져 있으므로 넣어주어야 할 것으로 보인다.


# drawings 테이블이 전반적인 수정이 필요하지 않을까?
게임을 진행하는데 기준이 되는 항목은 제시어이다. 하지만 현재의 테이블 컬럼은 제시어에 대한 내용이 담겨있지 않아서 해당 내용을 담아주어야 할 것으로 생각된다.
생각한 내용으로는 가장 처음으로 나온 제시어를 기준으로 게임이 진행되므로, 제시어를 기준삼아 테이블을 재 구성할 필요성이 보인다. 
필요한 기능은, 가장 처음의 제시어를 담고 / 해당 제시어를 말한 유저정보를 담고 / 해당 제시어에 대한 유저의 제시어를 담고 (유저의 정보도 담는다.) / 그 제시어의 그림을 담는다.(유저의 정보도 담는다.) / 최종적으로 그림 그림을 보고 어떤 그림인지 맞춰본다.(유저의 정보도 담는다.)

제세어 컬럼이랑 그림 컬럼을 나누는건??

questions
- id
- content : 질문내용
- connect : 원래 질문 id값 (그림의 question_primaryKey로 받아온다. 원래 질문일 경우 null)
- user_primaryKey
- room_primaryKey


drawings
- id
- content : 그림의 경로
- question_primaryKey 
- user_primaryKey
- room_primaryKey


# drawings 테이블에 room_id가 존재해야 할 듯?

그림을 그리고 저장할 때 하나의 그림이 한 row로 저장이 될 텐데 그렇다면 해당 방의 그림이라는 것을 파악하기 위해 생성되는 방의 고유값인 id 값을 저장하는 컬럼이 필요하지 않을까?

# 게임 이름을 정해야 하지 않을까?
게임 자체는 캐치마인드를 따온 것이 아니라 캐치마인드라는 이름을 쓰기도 그렇고
갈틱폰이라는 이름은 생소한 사람들이 많아 와닿지 않을 것 같다.
